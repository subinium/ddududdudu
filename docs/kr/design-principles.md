# 설계 원칙

## 범위

이 문서는 `ddudu`의 고수준 설계 철학을 다룬다.

나머지 문서들은 개별 서브시스템을 설명한다.
이 파일은 서브시스템 설계 결정의 바탕이 되는 세계관을 설명한다.

## 1. Harness가 제품이다

Harness 없는 모델은 대부분 텍스트 생성기다.

Harness가 모델에게 주는 것들:

- tool
- 지속적인 state
- 연속성
- 검증
- 위임
- 운영 경계

그 말은 유용한 설계 단위가 "prompt"만이 아니라는 뜻이다.
prompt, tool, state, 정책, 운영자 가시성의 조합이 설계 단위다.

## 2. 성능은 대부분 시스템 문제다

사람들이 AI 코딩 도구가 "더 잘 작동한다"고 말할 때, 그 의미는 여러 가지일 수 있다:

- 유효한 결과에 더 빨리 도달한다
- 재시도를 덜 한다
- token을 덜 낭비한다
- context를 덜 잃는다
- 실패에서 복구하기 쉽다

그 이득은 모델만이 아니라 harness에서 오는 경우가 많다.

실제로 그 의미는 종종:

- 전체 저장소 스냅샷 비용을 들이기 전에 명확한 리서치 작업을 먼저 라우팅한다
- 항목별 비교를 하나의 과부하된 리서처가 아닌 실제 병렬 워커로 분할한다
- 읽기 전용 리서치에는 쓰기 지향 격리나 검증 비용을 기본적으로 부과하지 않는다

따라서 성능은 "얼마나 많은 subagent가 존재하는가"가 아니다.
"harness가 불필요한 실행 오버헤드를 얼마나 줄이는가"다.

그것이 최적화 목표를 바꾼다:

- 작업 단위당 부트스트랩 비용을 줄인다
- 준비된 작업이 관련 없는 느린 작업을 기다리지 않고 계속되게 한다
- 검색, 모델, 쓰기, 검증 병목을 하나의 큐로 취급하는 대신 분리한다

## 3. Context는 축적이 아니라 선택해야 한다

Context engine은 최대 포함을 최적화해서는 안 된다.
신호 밀도를 최적화해야 한다.

그 의미:

- 전체 파일보다 관련 파일 범위
- transcript 재생보다 타입화된 artifact
- memory 덤프보다 목적 인식 memory 선택
- 편집이 활성화되면 변경된 파일 우선 검색
- 원시 웹 노이즈보다 읽기 쉬운 추출

## 4. State는 Harness에 속해야 한다

Provider runtime은 실행 엔진이다.
작업의 유일한 지속 기록이 되어서는 안 된다.

따라서 `ddudu`는:

- 정규 session state를 harness 소유의 진실로 취급한다
- provider session을 runtime 특정 실행 state로 취급한다

이를 통해 provider, mode, 워커가 변경되더라도 시스템이 연속성을 유지할 수 있다.

## 5. 위임은 압축 전략이다

위임은 주로 더 많은 agent를 갖는 것에 관한 것이 아니다.
격리와 전문화를 사용해 context 부하를 줄이는 것에 관한 것이다.

위임은 다음을 충족할 때만 유용하다:

- 작업을 경계 짓는다
- 예상 결과물을 정의한다
- 명시적인 성공 기준을 포함한다
- 단순한 텍스트가 아닌 구조화된 결과를 반환한다

그 말은 분해 자체가 중요하다는 뜻이기도 하다.
사용자가 `A/B/C` 리서치를 요청하면, 유용한 분할은 보통 추상적인 역할 레이블만이 아니라 주제별이다.

## 6. 가장 작은 실행 단위를 선호한다

Harness는 즉시 시작할 수 있는 가장 작은 작업 단위를 선호해야 한다.

그 의미:

- 하나의 워커가 지금 시작할 수 있을 때 관리형 오케스트레이션 전에 직접 실행
- 하나의 일관된 소유자가 다음 단계를 수행할 수 있을 때 전체 팀 실행 전에 위임
- 독립적인 비교 작업을 위한 주제 샤드
- 최종 답변 하나를 기다리기보다 좁은 diff와 점진적인 노트를 선호

핵심은 단순히 낮은 비용이 아니다.
실제 진행의 더 빠른 축적이다.

## 7. 실행은 필요한 만큼만 무거워야 한다

가장 작은 유용한 단위가 식별되면, harness는 여전히 모든 작업에 동일한 비용을 지불하는 것을 피해야 한다.

`ddudu`는 정책 기반 실행 가중치를 적용해야 한다:

- 전체 context 전에 최소 context
- 안전할 때 격리된 workspace 전에 공유 workspace
- 충분할 때 경량 검증 전에 검증 없음
- 작업이 주로 읽기 전용일 때 관리형 멀티 에이전트 오케스트레이션 전에 경량 팬아웃

이것이 여기서 "빠름"과 "안전함"이 반대가 아닌 이유다.
목표는 여전히 신뢰할 수 있는 가장 작은 runtime을 적용하는 것이다.

## 8. 병렬성은 처리량을 높여야지 의식을 늘려서는 안 된다

병렬성은 작업이 실제로 독립적일 때 유용하다.

그 의미:

- 독립적인 비교 작업을 위한 주제 팬아웃
- 읽기와 scout는 많이, 쓰기는 제한적으로
- 의존성이 해소될 때 연속 스케줄링
- 샤드가 완료되는 즉시 부분 결과
- 실제 가치를 추가하는 곳에서만 최종 합성 또는 검증

시스템은 기본적으로 모든 작은 샤드에 대해 완전히 관리되는 subagent를 만들어서는 안 된다.
그것은 종종 처리량보다 조정 비용을 더 많이 증가시킨다.

## 9. 검증은 경계에 위치해야 한다

시스템은 생성 후 검증을 있으면 좋은 것으로 취급해서는 안 된다.
하지만 모든 샤드에 동일한 검증 비용을 지불해서도 안 된다.

진지한 harness는 다음과 같은 루프를 중심으로 구축되어야 한다:

1. 실행
2. 검증
3. 복구
4. 에스컬레이션
5. 적용 또는 보고

중요한 뉘앙스는 루프가 어디에 위치하는가다.

- 쓰기 가능한 워커는 종종 적용 전에 검증이 필요하다
- 검토 및 배포 경계는 종종 읽기 전용 샤드보다 더 강한 검사가 필요하다
- 순수 리서치나 좁은 스카우트는 종종 전체 저장소 검증이 아니라 증거 품질만 필요하다

이것은 모든 워커를 가장 무거운 경로로 강제하지 않으면서 그럴듯한 작업보다 완료된 작업 쪽으로 시스템을 편향시킨다.

## 10. 신뢰는 Runtime 정책에 있어야 한다

Harness가 다음을 할 수 있게 되면 자연어 안전 지침만으로는 충분하지 않다:

- 셸 명령 실행
- 시크릿 접근
- 네트워크 호출
- MCP tool 호출
- workspace 수정

따라서 신뢰에는 다음이 필요하다:

- 명시적인 정책
- 명시적인 위험 분류
- 관찰 가능한 승인 경계
- runtime 강제

## 11. 페르소나는 인터페이스이지 아키텍처가 아니다

Mode, 전문가 레이블, agent 이름은 유용한 운영자 어포던스다.
실제 실행 모델이 아니다.

실제 모델은 다음과 같은 정책에 있어야 한다:

- 얼마나 많은 context를 로드할지
- 작업이 샤딩 가능한지
- 얼마나 많은 읽기와 쓰기를 허용할지
- 어떤 검증 티어를 요구할지
- 어떤 격리 경계를 강제할지

아키텍처가 페르소나 레이블에 너무 많이 의존하면, 성능 튜닝과 정확성 모두 어려워진다.

## 12. 사용자 질문은 구조화된 State여야 한다

Harness가 운영자에게 모호함을 해결하거나, 위험 경계를 선택하거나, 위험한 작업을 확인하도록 요청할 수 있게 되면, 그 질문들은 더 이상 단순한 텍스트가 아니다.

실행 state가 된다.

그 말은 진지한 harness가 다음을 모델링해야 한다는 뜻이다:

- 질문 종류: input, confirm, select, number, path 등
- 선택 메타데이터: 권장 또는 위험한 옵션
- 기본값과 유효성 검사
- 답변 출처: 선택된 옵션 vs 직접 입력

사용자 질문이 단순 문자열에 불과하다면, UI가 일관성 없어지고 정책 prompt를 신뢰하기 어려워진다.

## 13. UI는 제어 평면의 일부다

Harness가 다음을 지원하게 되면:

- 분리된 job
- 큐에 쌓인 작업
- 위임된 워커
- 복구 루프
- 백그라운드 실행

인터페이스는 장식적이기를 멈춘다.

운영자가 다음을 이해할 수 있는지를 결정하기 때문에 신뢰 모델의 일부가 된다:

- 무엇이 실행 중인지
- 무엇이 차단되어 있는지
- 무엇이 이미 통과했는지
- 누가 무엇을 소유하는지

## 14. ddudu는 Provider 워커 주변의 제어 평면이다

`ddudu`는 공식 도구들이 약하다는 가정 위에 만들어지지 않았다.

Provider runtime 위에 여전히 유용한 설계 공간이 있다는 가정 위에 만들어졌다:

- 커스텀 오케스트레이션
- 더 엄격한 context 규율
- 타입화된 핸드오프
- 더 강한 검증 루프
- 다른 신뢰 정책
- 다른 운영자 표면

## 설계 규칙

Harness를 소프트웨어 작업을 위한 운영 체제로 취급하라:
가장 작은 실행 단위를 빠르게 시작하고, 가시적인 진행을 축적하고, 운영자 입력이 필요할 때 명시적인 구조화된 질문을 하고, 처리량이나 신뢰를 높일 때만 더 무거운 오케스트레이션을 추가하라.
