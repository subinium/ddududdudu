# ddudu 기술 노트

이 디렉토리는 `ddudu`의 시스템 설계를 문서화한다.

최상위 [README](../../README.md)는 제품 관점에서 작성되어 있다:

- 사용 가능한 기능
- 설치 방법
- 명령어와 단축키
- 지원하는 워크플로우

이 디렉토리는 아키텍처 관점의 레이어다:

- coding harness가 실제로 어떤 문제를 해결하는지
- `ddudu`가 그 문제를 어떻게 분해하는지
- 어떤 트레이드오프가 어느 서브시스템에 속하는지
- 성능, 신뢰, 운영자 확신이 주로 어디서 오는지
- 지속적인 지식을 어떻게 선별하고, 승격하고, 검색해야 하는지

## 이 문서들이 존재하는 이유

대부분의 AI 코딩 시스템은 모델이 핵심 결과물이고 harness는 얇은 래퍼에 불과한 것처럼 설명된다.

그 관점은 잘못됐다.

실제로 harness가 결정하는 것들:

- 어떤 state가 살아남는지
- 어떤 context가 선택되는지
- tool이 어떻게 제어되는지
- 언제 작업이 위임되는지
- 실패가 어떻게 복구되는지
- 시스템이 실행 중일 때 운영자가 무엇을 신뢰할 수 있는지

이 문서들은 그 설계 결정들을 명시적으로 드러내기 위해 존재한다.

## 설계 입장

`ddudu`는 몇 가지 강한 입장을 중심으로 만들어졌다:

1. harness는 prompt만큼 중요하다
2. state는 transcript보다 내구성이 높다
3. context 품질이 context 크기보다 중요하다
4. 가장 작은 실행 단위를 먼저 시작해야 한다
5. 검증은 작업이 깨지거나, 완료되거나, 배포되는 경계에 위치해야 한다
6. 신뢰는 언어로 요청하는 것이 아니라 runtime 정책으로 강제해야 한다
7. 실행은 작업이 요구하는 만큼만 무거워야 한다
8. 병렬성은 더 많은 agent 의식을 만드는 것이 아니라 정책으로 제어해야 한다
9. UI는 장식용 껍데기가 아니라 제어 평면의 일부다
10. 대화형 질문은 임시 터미널 텍스트가 아니라 모델링된 state여야 한다

## 현재 구현 테마

현재 `ddudu` 구현은 몇 가지 실용적인 선택에 의해 특히 형성되어 있다:

- 요청 실행, 라우팅, 워크플로우 state, 팀 오케스트레이션, 백그라운드 생명주기가 이제 하나의 컨트롤러 덩어리 대신 별도의 runtime 경계 뒤에 위치한다
- 경량 리서치 팬아웃이 이제 모든 병렬 요청을 관리형 팀 실행인 척하는 대신 자체 runtime 경계를 갖는다
- 순수 외부 리서치는 경량 context 경로를 사용하며, prompt에 여러 주제가 명시되면 여러 읽기 전용 워커로 팬아웃할 수 있다
- 혼합 구현 작업은 이제 관리형 팀 실행으로 에스컬레이션하기 전에 직접 또는 위임 실행을 선호한다
- provider, 검색, 쓰기, 검증 압력이 이제 관련 없는 대기로 취급되는 대신 하나의 scheduler 모델을 공유한다
- 병렬 팀 실행이 이제 전체 라운드 배리어를 기다리는 대신 의존성이 해소되는 대로 계속된다
- ask-user prompt가 이제 일반 문자열 prompt 대신 질문 종류, 기본값, 유효성 검사, 답변 출처를 가진 구조화된 runtime state다
- session과 운영자 설정은 기본적으로 전역 우선이며, 프로젝트 `.ddudu/` config는 명시적인 오버라이드 레이어다
- 프로젝트 instruction은 `.ddudu/DDUDU.md`와 함께 크로스 툴 표준인 `AGENTS.md`를 지원하므로, 한 번 작성한 규칙이 Claude Code, opencode, Codex CLI, Amp에서 모두 적용된다
- 운영자 표면은 이제 중복된 runtime 세부 정보보다 실행 state, todo 소유권, 워커 가시성, 명시적인 ask-user 선택을 우선시한다
- TUI가 영역별 시각 인코딩을 사용한다 — 사이드바는 한 단계 어둡고, 메인은 중립적이고, 컴포저는 한 단계 밝다 — 텍스트를 읽기 전에 공간적으로 위치를 파악할 수 있다
- result augmentation이 tool 호출 후 행동 넛지를 주입한다 (검증 리마인더, 진단 힌트). 모델이 operator 개입 없이 스스로 교정하도록 유도한다
- API 호출에 지수 백오프와 jitter를 적용한 자동 재시도가 추가됐다. 에러 분류가 일시적 실패, 인증 오류, 치명적 오류를 구분한다
- 컴포저가 Arrow-Up/Down으로 prompt 히스토리를 불러올 수 있다. 중복 제거와 임시 저장(stash)을 지원하며, 반복 워크플로우를 빠르게 만든다

## 이 문서들이 아닌 것

이것들은 마케팅 노트도 아니고 명령어 레퍼런스도 아니다.

다음 질문들에 답하기 위해 작성됐다:

- 이 서브시스템이 왜 존재하는지
- 무엇을 소유해야 하는지
- 무엇을 소유하면 안 되는지
- 어디서 실패하는 경향이 있는지
- `ddudu`가 현재 무엇을 하는지

그래서 이 폴더는 완성된 제품 매뉴얼보다 살아있는 아키텍처 노트북에 가깝다.
