# Trust And Sandbox

## 범위

이 문서는 coding harness에서의 실행 trust, 정책, 그리고 enforcement 경계를 다룬다.

## 철학

저장소를 수정하거나, 셸 명령을 실행하거나, 네트워크에 접근하거나, 외부 도구를 호출할 수 있는 harness는 자연어 주의 문구보다 훨씬 강력한 trust 모델이 필요하다.

핵심 철학은:

- 리스크는 가시적이어야 한다
- trust는 설정 가능해야 한다
- 정책은 실행 전에 적용되어야 한다
- 문서는 runtime이 실제로 제공하지 않는 보장을 암시해서는 안 된다

## 정책 레이어 vs Enforcement 레이어

이 둘은 다른 것이다.

| 레이어 | 목적 |
| --- | --- |
| Trust policy 레이어 | 리스크 분류, 허용, 확인 요청, 또는 거부 |
| Sandbox enforcement 레이어 | 파일시스템, 프로세스, 또는 네트워크 접근을 기술적으로 제한 |

많은 시스템이 첫 번째를 구현하면서 두 번째인 것처럼 느슨하게 설명한다.
이 둘은 명확히 구분해야 한다.

## 리스크 표면

coding harness에서 주요 리스크 표면은 보통:

- 셸 실행
- 네트워크 접근
- 시크릿 및 자격증명 접근
- MCP 서버와 외부 도구
- 위임된 worker
- 메인 workspace에 patch 적용

이 표면들은 영향 범위와 되돌릴 수 있는 정도가 각각 다르다.

## 좋은 Trust 설계

### 1. 리스크 가시성

harness는 어떤 액션이:

- 로컬인지
- 네트워크를 사용하는지
- 시크릿에 인접한지
- 파괴적인지
- 위임된 것인지
- 외부인지

를 알아야 한다.

### 2. 설정 가능한 trust

유용한 제어 지점:

- 전역 permission profile
- 도구별 정책
- 호스트별 네트워크 정책
- 보호된 경로
- 보호된 환경 변수
- MCP 서버별 trust 등급

### 3. Runtime enforcement

trust는 모델이 어떤 문단을 기억하는 것에 의존해서는 안 된다.

runtime은 다음을 강제해야 한다:

- 하드 차단된 셸 패턴
- 명시적 승인 경계
- 호스트 수준 trust
- 시크릿 경계

### 4. 계층화된 정책

유용한 질문은 "이것이 허용되는가?"만이 아니다.
다음도 물어야 한다:

- 왜 허용되는가?
- 어떤 profile 하에서?
- 어떤 도구 정책 때문에?
- 어떤 호스트 규칙 때문에?
- 어떤 MCP trust 설정 때문에?

### 5. 구조화된 승인

승인 프롬프트는 enforcement의 일부이지, 겉치레 UX가 아니다.

좋은 harness는 다음을 표시해야 한다:

- 액션 요약
- 현재 permission profile
- 관련 trust boundary 이유
- 기본 거부 경로
- 어떤 선택이 권장되거나 위험한지

이것은 operator가 더 허용적인 모드로 전환하거나 위험한 도구 호출을 허용할 때 특히 중요하다.

## MCP Trust

MCP는 first-class trust 표면으로 취급해야 한다.

권장 등급:

- `trusted`
- `ask`
- `deny`

이것 없이는 MCP가 안전하지 않거나 사용 불가능해지는 경향이 있다.

## Worktree vs Sandbox

Git worktree는 다음에 도움이 된다:

- patch 격리
- 충돌 감소
- 더 안전한 위임 실행

Git worktree는 다음을 해결하지 못한다:

- 시크릿 격리
- 호스트 프로세스 격리
- 네트워크 격리

이것은 운영상의 격리이지, 완전한 sandbox가 아니다.

## ddudu 구현 노트

현재 `ddudu`는 다음을 지원한다:

- 하드 차단된 셸 패턴
- 리스크 분류
- permission profile
- 도구별 정책
- 호스트 기반 네트워크 trust
- 보호된 시크릿 경로와 환경 변수
- MCP trust 등급
- 위험한 도구 호출에 대한 구조화된 승인 프롬프트
- `permissionless`로 전환하기 전 명시적 확인

아직 제공하지 않는 것은 컨테이너나 VM 격리 같은 완전한 프로세스 수준 sandbox다.

이 한계는 명확히 문서화하기 위해 의도적으로 남겨둔 것이다.

## 설계 원칙

harness가 operator를 대신해 행동할 만큼 강력하다면, trust 정책은 명시적 경계와 관찰 가능한 결정을 갖춘 first-class 서브시스템이어야 한다.
